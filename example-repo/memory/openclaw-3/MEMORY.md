## 审查工作流
```
宇的需求 (00_input/requirement.md)
    ↓
1号产出 → 10_architecture/project-brief.md + 20_tasks/TASK-xxx/task-card.md
    ↓
2号执行 → 30_execution/ 执行产物
    ↓
3号审查 → 40_review/review-YYYYMMDD-HHMM.md
    ↓
报告影响1号和2号的下一轮改进
```

## 审查原则
- **先跟着1号的思路走**，不先去否定1号
- **方向性问题必须指出来**（需求理解偏了、执行结果和目标差太远）
- 审查重点：需求理解 → 架构设计 → 任务拆解 → 执行质量
- 报告是给1号和2号看的，驱动他们改进

## 审查清单
1. 原始需求是什么？（读 00_input/）
2. 1号对需求的理解方向对不对？（读 10_architecture/）
3. 任务拆解是否合理可执行？（读 20_tasks/）
4. 2号是否按任务卡执行？产出质量如何？（读 30_execution/）
5. 方向/漏项/重复/过度发挥问题？
