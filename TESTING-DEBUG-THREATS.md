# Heart-Beat-War 测试、调试与威胁模型

这份文档解决三件事：

- 怎么验证系统真的能跑
- 出问题时按什么顺序查
- 这个三机协作系统最容易被什么问题打穿

## 1. 测试目标

不是只测“模板文件存在”，而是要测下面 4 件事：

1. 角色有没有串位
2. 仓库状态流能不能闭环
3. heartbeat 能不能触发实际动作
4. 渠道消息和仓库协作会不会互相打架

## 2. 推荐测试层级

### L1：静态模板测试

目标：

- 检查模板是否包含必填字段
- 检查引用关系是否完整

至少检查：

- `task-card.md` 是否引用 `execution-checklist.md` 和 `test-plan.md`
- `status-report.md` 是否包含批次编号、检查结果、阻塞、下一步建议
- `review-report.md` 是否能输出 `CONTINUE / REWORK / CLOSE_*`

通过标准：

- 1 号根据模板就能发出一个任务包

### L2：单机角色测试

目标：

- 每个角色单独启动时能否只做自己该做的事

至少检查：

- 1 号不会写 `30_execution/`
- 2 号不会改 `10_architecture/`
- 3 号不会改 `30_execution/`

通过标准：

- 根目录 `AGENTS.md` 和 workspace 路径能唯一决定角色

### L3：双机链路测试

目标：

- 1 号到 2 号
- 2 号到 3 号

测试步骤：

1. 1 号发一个最小任务
2. 2 号发现任务并回写 `HANDOFF.md` 与 `STATUS-REPORT.md`
3. 3 号发现新结果并写 `40_review/`

通过标准：

- 三步都能在共享仓库中看到结果

### L4：全链路集成测试

目标：

- 从用户输入到 review 结论完整闭环

测试步骤：

1. 在 `00_input/` 放一条简单需求
2. 1 号产出架构和任务包
3. 2 号执行一个小步骤
4. 3 号审查
5. 1 号基于审查发第二轮小步骤

通过标准：

- 至少出现两轮批次流转

### L5：渠道集成测试

目标：

- 飞书或微信消息能和仓库动作对应上

至少检查：

- 在 DM 下指令，1 号能正确 intake
- 在目标群内，消息目标能被正确解析
- heartbeat 输出能送到预期会话，而不是掉到错误对话

通过标准：

- 你发一条消息后，能在仓库或群里看到对应动作结果

### L6：故障恢复测试

目标：

- 验证系统在常见异常下能恢复

至少测试：

- 杀掉 gateway 后能否恢复
- git 冲突后是否会重复执行
- 错角色配置是否能被发现
- 旧 task 残留是否会引发重复消费

## 3. 推荐测试矩阵

| 编号 | 用例 | 重点 | 通过标准 |
|---|---|---|---|
| T-01 | 角色检测 | workspace 与 `AGENTS.md` 一致 | 三台角色不串位 |
| T-02 | git 写权限 | pull / push 正常 | 三台都能提交 |
| T-03 | 任务下发 | 1 号能创建任务包 | `20_tasks/TASK-XXX/` 完整 |
| T-04 | 执行回写 | 2 号写状态与交接 | `HANDOFF.md` 与 `STATUS-REPORT.md` 存在 |
| T-05 | 审查结论 | 3 号给出明确结论 | 能判断继续/返工/关闭 |
| T-06 | 批次编号 | 三方引用同一编号 | 无编号漂移 |
| T-07 | heartbeat | 周期触发实际动作 | 不只回 `HEARTBEAT_OK` |
| T-08 | 群聊路由 | 消息送到正确会话 | 不串群、不串 DM |
| T-09 | 幂等性 | 重复扫描不重复执行 | 同批次不重复写产物 |
| T-10 | 恢复能力 | 进程异常可恢复 | 恢复后继续工作 |

## 4. 推荐的验收样本项目

不要用大项目做第一轮验收。

推荐先用这种最小样本：

- 需求：写一份 300 字 landing page 草稿
- 1 号：拆成 2 个小步骤
- 2 号：先只完成标题与价值主张
- 3 号：审内容是否对齐需求

好处：

- 无需复杂代码
- 容易判断对错
- 能快速暴露链路问题

## 5. 调试总原则

调试顺序永远是：

1. 先查角色
2. 再查 workspace
3. 再查 git
4. 再查 heartbeat
5. 再查渠道
6. 最后才查 prompt 和模型行为

如果顺序反过来，极容易把“配置问题”误判成“模型理解问题”。

## 6. 常见故障与排查顺序

### 症状 A：1 号发了任务，2 号不干活

先查：

1. 2 号 heartbeat 是否真的在跑
2. 2 号 workspace 是否指向正确 repo
3. 2 号 `AGENTS.md` 是否真的是执行者角色
4. `20_tasks/` 是否用了 2 号看得懂的目录结构
5. git 是否 pull 到最新

高频原因：

- 2 号角色串成 3 号
- heartbeat 只回 `HEARTBEAT_OK`
- 任务卡位置和 heartbeat 扫描规则不一致

### 症状 B：2 号执行了，但 3 号不审

先查：

1. 3 号 heartbeat 是否在跑
2. `30_execution/` 是否按预期路径产出
3. `STATUS-REPORT.md` 是否存在
4. 3 号是否能定位到对应 task 和批次

高频原因：

- 3 号只会看平铺目录，不会看任务子目录
- 产物路径和文档约定不一致
- 3 号角色或 workspace 串位

### 症状 C：群里看不到消息

先查：

1. bot 是否真的在目标群
2. 群是否加入 allowlist
3. target 是不是还停留在旧会话
4. DM 和群聊策略是不是冲突
5. 是否其实只有 1 号接了飞书，2 号和 3 号没有

高频原因：

- 你以为三机都在飞书，实际只有 1 号在飞书
- target 绑定在旧私聊
- groupPolicy 没开放

补充说明：

- 飞书群聊里看不到群 ID 是常见现象
- 如果系统支持 `target = "last"` 或等效“最近会话”模式，先在目标群里和 bot 对话一次，往往就不必手填群 ID
- 如果必须做 allowlist，就从飞书事件回调日志、OpenClaw 日志或 webhook payload 里抓取真实 `chat_id`

### 症状 C-2：你说一句话，三个 bot 一起抢答

先查：

1. 2号和 3号是否也在直接消费用户聊天
2. 2号和 3号的行为说明里，是否明确规定“不能直接接新需求”
3. 群聊是否被当成协作总线，而不是只当用户入口

高频原因：

- 没有群聊路由规则
- 1号、2号、3号都把群聊当成“可以直接派工”的入口
- 仓库协作链和聊天入口链混在一起了

### 症状 D：重复执行

先查：

1. heartbeat 间隔和任务幂等判断
2. `30_execution/` 是否包含对应批次标识
3. 2 号是否判断过“本批次已执行”

高频原因：

- 没有批次编号
- 没有“已执行”判定
- task 长时间不删除或不关闭

### 症状 E：明明配置对了，但行为还是怪

先查：

1. 根目录 `AGENTS.md`
2. 根目录 `HEARTBEAT.md`
3. workspace 根目录是否混入别的项目文件
4. 历史 memory 是否把旧规则带进来了

高频原因：

- 不是模型不行，而是旧上下文污染了当前角色

## 7. 推荐 debug 命令清单

下面这些命令建议每台机器都能熟练跑：

### 基础身份检查

```bash
whoami
hostname
pwd
```

### 版本检查

```bash
node -v
pnpm -v
openclaw --version
```

### workspace 检查

```bash
ls -la ~/.openclaw
find ~/.openclaw -maxdepth 2 -type d | sort
```

### 配置检查

```bash
cat ~/.openclaw/openclaw.json
cat AGENTS.md
cat HEARTBEAT.md
```

### 进程检查

```bash
ps -ef | grep -i openclaw | grep -v grep
```

### git 检查

```bash
git status
git remote -v
git branch --show-current
git pull --rebase
```

### heartbeat 相关检查

```bash
cat ~/.openclaw/cron/jobs.json
```

如果有日志文件，也要检查最近日志中的：

- heartbeat started
- delivery failed
- channel target not found
- auth failed

## 8. 潜在威胁模型

### 威胁 1：角色漂移

表现：

- 2 号写 review
- 3 号去执行任务

根因：

- workspace 配错
- `AGENTS.md` 放错
- 旧 memory 污染

缓解：

- 每台机器单独 workspace
- 根目录放显式角色标识
- 开工先跑角色检测

### 威胁 2：渠道漂移

表现：

- 文档写飞书，实际通道是微信
- 消息发到错误对话

缓解：

- 配置文件和部署文档保持一致
- 单飞书群聊模式下不要混用微信
- 每次改通道后做一次消息配对测试

### 威胁 2.1：群聊扇出和串任务

表现：

- 用户一句话触发 1号、2号、3号同时响应
- 2号和 3号绕过仓库直接接了任务

缓解：

- 1号是唯一任务入口
- 2号和 3号只认仓库任务，不认用户聊天派工
- 群里对外播报可以有多个 bot，但派工入口只能有一个

### 威胁 3：仓库不是唯一真相源

表现：

- 关键信息只在聊天里
- 仓库里看不出为什么发这个任务

缓解：

- 所有可执行结论都落文件
- 1 号不依赖聊天记忆派工

### 威胁 4：重复消费任务

表现：

- 2 号重复做同一个步骤

缓解：

- 任务与批次编号明确
- 2 号执行前检查已有产物
- 3 号明确关闭批次或任务

### 威胁 5：群消息泄漏或串群

表现：

- 内部进度发到错误群
- 在不该公开的群里播报

缓解：

- 使用 allowlist
- 上线前先在测试群验证
- 生产群与实验群分离

### 威胁 6：敏感信息泄漏

表现：

- API key、token、群 ID 被写进仓库

缓解：

- 仓库只放模板，不放真实密钥
- 真实值只存服务器本地配置
- 对日志和 handoff 做脱敏

### 威胁 7：1 号成为单点故障

表现：

- 1 号挂了，整个流程没人继续

缓解：

- 关键状态全部写入仓库
- 2 号和 3 号的 heartbeat 不依赖 1 号在线
- 保留人工接管手册

## 9. 生产前必须补的保护措施

至少补下面这些：

- 批次编号规范
- task 目录命名规范
- 幂等判断规则
- 群聊 allowlist
- 测试群与生产群分离
- 敏感配置本地化
- 错误日志留存
- 一键自检脚本

## 10. 什么时候才算“可上线”

同时满足下面条件，才算接近可用：

1. 完成至少一次全链路双轮演练
2. 三台机器角色不串位
3. 三台机器 git 正常
4. 至少一个渠道能稳定接收和回发
5. 至少验证一次进程异常恢复
6. 至少验证一次重复执行保护
7. 至少验证一次群聊路由正确

如果这些还没全过，就不要把它当成“我在飞书里发一句话，系统就能稳定自动协作”的生产系统。
